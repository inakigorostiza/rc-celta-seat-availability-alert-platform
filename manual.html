01
Introduction

The problem, the solution, and why it matters.

A fan wants to see RC Celta play. The section is sold out. Days later a seat frees up — but the ticket office has no record that this fan wanted it. That missing memory is silent revenue lost.

❌
The problem

Invisible lost demand

A significant share of tickets that free up (refunds, failed payments, released pre-allocations) never reach the fans who wanted them. The sale evaporates because nobody told them.

✅
The solution

Automated recovery

Capture interest while inventory is sold out, continuously watch availability in ONEBOX, and send a personalised email with a purchase deep-link the moment the requested section has seats again.

💡

Key takeaway

Recovering lost demand is a timing game. The value isn't in owning the list — it's in reaching the customer the moment the problem that stopped them from buying has been solved.

04
Credentials

Four credentials, none paid on the starter plan.

This is the minimum setup. Every key either has a generous free tier or is free-forever. Budget 30 minutes to have them all lined up before you start building.

What you need

client_secret
32-character string that identifies your channel in ONEBOX.
channel_id
Integer that identifies the channel inside the organisation (e.g. 2287).

Where to get it

Provided by ONEBOX's technical team. Ask your account contact.

⚠️

Watch out

The client_secret grants access to the whole channel catalog. Never embed it in the frontend or commit it to a public repo. Keep it in your platform's secret manager.

What you need

API key
Token to call the Marketing API from n8n and the automation layer.
Audience ID
Identifier of the audience (list) where your contacts will live.
Server prefix
Suffix of the API key (e.g. ...-us1 → us1). Tells you which datacenter your account lives on.

Where to find it

API key: Profile → Account & billing → Extras → API keys

Audience ID: Audience → Settings → Audience name and defaults

What you need

API key
Token to call the Transactional API from n8n.
Verified sending domain
A domain you control with DKIM + SPF records published and verified in Mandrill.

Where to find it

API key: Mandrill → Settings → SMTP & API Info → New API key

Domains: Settings → Sending Domains
⚠️

Watch out

Without a verified domain, Mandrill rejects every send with unsigned. It's the number-one error beginners hit. Publish the TXT records before you wire up the flow.

What you need

Measurement ID
The G-XXXXXXX identifier of the web data stream.

Where to find it

GA4 → Admin → Data streams → Web → your stream

This one is safe to expose in public HTML — it's designed to be visible. GA4 distinguishes sessions via cookies and URL parameters, not by the ID itself.

06
ONEBOX

The ticketing API.

ONEBOX TDS is the platform where RC Celta sells tickets online. It exposes a professional REST API that we use exclusively in read mode — we never modify data in ONEBOX.

Data model (with names a marketer understands)

Event (e.g. «Celta – Athletic · Matchday 29») └── Session (a specific performance, with date and time) ├── Price types (General, Junior, Season ticket…) └── Sections (Pista B1, Río Alto, Marcador…) └── availability = N free seats
💡

Key takeaway

To the fan we speak match and section. The internal IDs (session_id, grada_id) are for the machines.

The three endpoints we use

POST /oauth/token Request an access token (valid 12 hours)
GET /catalog-api/v1/sessions List every match in the channel
GET /catalog-api/v1/sessions/{id}/availability Per-section availability for a match

The purchase deep-link

When we send the alert email, the Buy ticket button points at a URL that opens ONEBOX directly on the seat map, with every UTM GA4 needs to attribute the conversion:

https://tickets.oneboxtds.com/rccelta/select/{sessionId}?    viewCode=V_blockmap    &utm_source=avisame    &utm_medium=email    &utm_campaign=recovery-app    &utm_content=event-{eventId}-grada-{gradaId}

viewCode=V_blockmap is functional — it opens the seat map directly. The four utm_* parameters let GA4 attribute any resulting purchase to the exact email that drove the click.

Try it yourself · Test ONEBOX in a webpage

Use Claude Code (see §05) to go from zero to a working API test harness in under an hour. These are the exact five prompts that produced the four live webpages shipping with this case study — copy them into your Claude Code panel one by one.

⚙️

Before you start

You need: ONEBOX PRE credentials (client_secret + channel_id), Node.js 18+, VS Code with the Claude Code extension, and an empty folder opened as a workspace. Create a .env file in the folder with your two ONEBOX values.

01
Authenticate against ONEBOX → test-onebox.js
Create a Node.js script called test-onebox.js that authenticates against the ONEBOX PRE environment at https://api.oneboxtds.net/oauth/token using the seller-channel-client OAuth2 flow. Read ONEBOX_CLIENT_SECRET and ONEBOX_CHANNEL_ID from a .env file. Use native Node.js fetch (no npm install). On success, print the token expiry and granted scopes. On failure, print the HTTP status and response body and exit 1.

What you should see: Claude creates the script, runs it, and prints ✔ token acquired, expires_in=43200s, scope=api-catalog-all api-distribution-all…. If it fails, Claude will read the error and ask you to double-check the credentials.

02
Make your first real API call → extends test-onebox.js
Extend test-onebox.js so that after authentication it also GETs /catalog-api/v1/sessions with the bearer token. Print how many sessions came back (metadata.total), plus the first session's name and event name.

What you should see: ✔ 21 sessions · first: NUMERADO in * API NUMBERED EVENT. This proves authentication + catalog read end-to-end.

03
Render a live documentation page → generate-examples.js · examples.html
Create generate-examples.js that runs live calls to (1) /oauth/token, (2) /catalog-api/v1/sessions, and (3) /catalog-api/v1/sessions/{id}/availability for the first session. Render the results into a static examples.html page with three labelled sections — one per call — showing the request method/URL/headers and the JSON response. Use a dark theme with monospace code blocks. Redact the access_token and refresh_token in the output.

What you should see: open examples.html shows a clean three-section doc of live request/response examples. Safe to share with teammates — secrets are hidden.

04
Visualise the catalog as a card grid → generate-sessions.js · sessions.html
Create generate-sessions.js that fetches every session and its availability, then produces sessions.html — a responsive card grid. Each card shows the event image, event name, venue + city, start date in the venue timezone, price range, availability bar (green > 50%, amber > 15%, red below), price-type breakdown using ui_settings.color for each, sectors in a collapsible, and a Raw JSON toggle. Add a live-filter input at the top that matches by event or venue name.

What you should see: open sessions.html shows all 21 sessions as branded cards. Hover a card, expand a sector, type a venue name in the filter — all data is live from ONEBOX.

05
Build the branded signup landing → generate-landing.js · landing.html
Create generate-landing.js that builds landing.html for a single ONEBOX session (hardcode the session_id at the top of the file). The landing should mirror avisame-celta.netlify.app visually: sky-blue hero with RC Celta crest, big white "¡Avísame!" title, white card with Nombre / Apellidos / Email fields and a Grada dropdown populated from that session's sectors. On submit, validate the form and log the payload to the console for now.

What you should see: open landing.html shows a production-looking fan-facing form. Fill it in, submit, open DevTools console — the payload is there, ready to wire to Mailchimp in §08.

08
Mailchimp

The central list, segmentable without SQL.

Every registered fan lives here. Two distinct mechanisms organise their data: merge fields (their profile) and tags (their interest history). Each has different semantics, and that changes how we use them.

Merge fields Overwrite

The contact's "profile". Each field holds one value: the most recent one. Ideal for data that's by its nature unique per contact.

Example: Ana signs up for Pista B1 today. Tomorrow she signs up for Río Alto.

GRADA_NAME: Pista B1 → Río Alto
GRADA_ID: 224549 → 224562

Tags Stack

They preserve the full history. A contact can carry dozens of tags at once. Ideal for segmenting by interests or behaviour.

Same case: Ana signs up for Pista B1 and later for Río Alto.

grada:224549 (Pista B1)
grada:224562 (Río Alto)
event:4587
source:avisame
💡

Key takeaway

Use tags to segment. Use merge fields for the visible profile in the Mailchimp UI. They're complementary, not alternatives.

Typical segment: Sector 316 of Celta–Athletic opens

tag contains  grada:224515         AND
tag contains  event:4587           AND
tag NOT ∋     notified:event-4587-grada-224515

Marketing builds this segment in Mailchimp's UI. n8n consumes it to send the alert at most once per fan and section.

What we measure per send

Tracking enabled

Open rate · CTR

Filters in the panel

Tags per match + section

Watch out

The domain you send from (from_email) must be verified in Mandrill with DKIM + SPF. If it isn't, every send is rejected with reject_reason: unsigned. It's the first mistake everyone makes.

Legal & Privacy

GDPR · LOPDGDD · LSSI-CE.

Three regulations overlap in this project. All are mandatory if you operate in Spain. They're not "nice to have" — without compliance, the platform cannot go to production.

🇪🇺
GDPR (RGPD)
General Data Protection Regulation · EU 2016/679

EU regulation that sets the overall framework for how personal data must be collected, processed and stored. Applies to any processing of EU residents' data.

  • Free, specific, informed and unambiguous consent before collecting the email
  • Data minimisation — collect only what's strictly necessary
  • Purpose limitation — use data only for what was consented
  • Data-subject rights guaranteed contractually (access, rectification, erasure, portability…)
  • Record of Processing Activities documented
🇪🇸
LOPDGDD
Spanish Organic Law 3/2018

The Spanish adaptation of GDPR. Reinforces user rights and sets the national enforcement regime.

  • Minimum age 14 years to consent on one's own
  • Digital rights: portability, right-to-be-forgotten on social media, digital disconnection
  • Fines up to €20M or 4% of global turnover
  • Mandatory Data Protection Officer for large-scale processing
📧
LSSI-CE
Spanish Law 34/2002 · Information Society Services

Regulates electronic commercial communications and the use of cookies.

  • Prior, express opt-in for commercial emails (Art. 21)
  • Free and easy unsubscribe link in every communication
  • Clear identification of the sender
  • Blocking cookie banner for any non-strictly-necessary cookie (GA4 falls here)
  • Fines up to €150,000 for serious infringements
